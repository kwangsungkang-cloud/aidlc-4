# 테이블오더 서비스 — User Stories

분류: 사용자 여정 기반 | 수용 기준: Given/When/Then | 우선순위: P1/P2/P3

---

## 고객 여정: 입장 → 메뉴 조회 → 장바구니 → 주문 → 확인 → 분할 계산

---

### US-01: 테이블 태블릿 자동 로그인
**우선순위**: P1
**페르소나**: 디지털 친숙 고객, 디지털 비친숙 고객
**요구사항**: FR-01

> As a 고객,
> I want 테이블에 앉으면 별도 로그인 없이 바로 주문 화면을 볼 수 있기를,
> So that 대기 시간 없이 즉시 메뉴를 탐색하고 주문할 수 있다.

**Acceptance Criteria**:

1. Given 관리자가 태블릿에 매장 식별자, 테이블 번호, 테이블 비밀번호를 1회 설정했을 때,
   When 태블릿 브라우저에서 서비스에 접속하면,
   Then 저장된 정보로 자동 로그인되어 메뉴 화면이 표시된다.

2. Given 자동 로그인 정보가 로컬에 저장되어 있을 때,
   When 브라우저를 새로고침하면,
   Then 자동 로그인 상태가 유지되어 메뉴 화면이 다시 표시된다.

3. Given 자동 로그인 정보가 저장되어 있지 않을 때,
   When 태블릿에서 서비스에 접속하면,
   Then 초기 설정 화면(매장 식별자, 테이블 번호, 비밀번호 입력)이 표시된다.

4. Given 잘못된 로그인 정보가 저장되어 있을 때,
   When 자동 로그인이 실패하면,
   Then 에러 메시지와 함께 초기 설정 화면이 표시된다.

### US-01-1: 초기 접속 안내 화면
**우선순위**: P1
**페르소나**: 디지털 친숙 고객, 디지털 비친숙 고객
**요구사항**: FR-01-1

> As a 고객,
> I want 처음 접속했을 때 환영 메시지와 사용법 안내를 볼 수 있기를,
> So that 서비스 이용 방법을 쉽게 이해하고 자신 있게 주문을 시작할 수 있다.

**Acceptance Criteria**:

1. Given 고객이 자동 로그인에 성공했을 때,
   When 해당 세션에서 처음 접속하면,
   Then 매장명, 테이블 번호, 환영 메시지, 사용법 안내가 포함된 안내 화면이 표시된다.

2. Given 안내 화면이 표시되어 있을 때,
   When "주문 시작하기" 버튼을 누르면,
   Then 메뉴 화면으로 이동한다.

3. Given 세션 내에서 안내 화면을 이미 본 적이 있을 때,
   When 다시 서비스에 접속하면,
   Then 안내 화면 없이 바로 메뉴 화면이 표시된다.

4. Given 디지털 비친숙 고객이 안내 화면을 볼 때,
   When 사용법 안내를 확인하면,
   Then 큰 글씨와 아이콘으로 메뉴 탐색 → 장바구니 담기 → 주문하기 흐름이 설명된다.

### US-02: 메뉴 조회 및 탐색
**우선순위**: P1
**페르소나**: 디지털 친숙 고객, 디지털 비친숙 고객
**요구사항**: FR-02

> As a 고객,
> I want 카테고리별로 메뉴를 탐색하고 상세 정보를 확인할 수 있기를,
> So that 원하는 메뉴를 쉽게 찾아 선택할 수 있다.

**Acceptance Criteria**:

1. Given 고객이 메뉴 화면에 있을 때,
   When 화면이 로드되면,
   Then 카테고리 목록과 첫 번째 카테고리의 메뉴가 카드 형태로 표시된다.

2. Given 메뉴 목록이 표시되어 있을 때,
   When 다른 카테고리를 선택하면,
   Then 해당 카테고리의 메뉴만 표시된다.

3. Given 메뉴 카드가 표시되어 있을 때,
   When 메뉴 카드를 확인하면,
   Then 메뉴명, 가격, 설명, 이미지가 표시된다.

4. Given 디지털 비친숙 고객이 사용할 때,
   When 메뉴 화면을 조작하면,
   Then 모든 버튼이 최소 44x44px 이상이고 명확한 시각적 계층 구조를 가진다.

### US-03: 장바구니 관리
**우선순위**: P1
**페르소나**: 디지털 친숙 고객, 디지털 비친숙 고객
**요구사항**: FR-03

> As a 고객,
> I want 주문 전에 선택한 메뉴를 장바구니에서 확인하고 수정할 수 있기를,
> So that 실수 없이 원하는 메뉴와 수량으로 주문할 수 있다.

**Acceptance Criteria**:

1. Given 고객이 메뉴 화면에서 메뉴를 선택했을 때,
   When "담기" 버튼을 누르면,
   Then 해당 메뉴가 장바구니에 추가되고 장바구니 아이콘에 수량이 표시된다.

2. Given 장바구니에 메뉴가 있을 때,
   When 수량 증가/감소 버튼을 누르면,
   Then 수량이 변경되고 총 금액이 실시간으로 재계산된다.

3. Given 장바구니에 메뉴가 있을 때,
   When 특정 메뉴의 삭제 버튼을 누르면,
   Then 해당 메뉴가 장바구니에서 제거되고 총 금액이 재계산된다.

4. Given 장바구니에 메뉴가 있을 때,
   When "장바구니 비우기" 버튼을 누르면,
   Then 모든 메뉴가 제거되고 총 금액이 0원으로 표시된다.

5. Given 장바구니에 메뉴가 있을 때,
   When 브라우저를 새로고침하면,
   Then 장바구니 내용이 유지된다 (로컬 저장).

### US-04: 주문 생성
**우선순위**: P1
**페르소나**: 디지털 친숙 고객, 디지털 비친숙 고객
**요구사항**: FR-04

> As a 고객,
> I want 장바구니의 메뉴를 최종 확인 후 주문을 확정할 수 있기를,
> So that 정확한 주문이 매장에 전달된다.

**Acceptance Criteria**:

1. Given 장바구니에 1개 이상의 메뉴가 있을 때,
   When "주문하기" 버튼을 누르면,
   Then 주문 내역 최종 확인 화면이 표시된다 (메뉴명, 수량, 단가, 총 금액).

2. Given 주문 확인 화면에서,
   When "주문 확정" 버튼을 누르면,
   Then 주문이 서버에 전송되고 주문 번호가 표시된다.

3. Given 주문이 성공적으로 생성되었을 때,
   When 주문 번호가 표시된 후,
   Then 장바구니가 자동으로 비워지고 메뉴 화면으로 자동 리다이렉트된다.

4. Given 주문 전송 중 오류가 발생했을 때,
   When 서버 응답이 실패하면,
   Then 에러 메시지가 표시되고 장바구니 내용은 유지된다.

5. Given 주문이 생성될 때,
   When 서버에 주문 데이터가 전송되면,
   Then 매장 ID, 테이블 ID, 메뉴 목록(메뉴명, 수량, 단가), 총 금액, 세션 ID가 포함된다.

---

### US-05: 분할 계산
**우선순위**: P2
**페르소나**: 디지털 친숙 고객
**요구사항**: FR-04-1

> As a 그룹으로 식사하는 고객,
> I want 총 주문 금액을 메뉴별 또는 인원별로 나눠서 계산할 수 있기를,
> So that 각자 부담할 금액을 쉽게 확인하고 매장에서 별도로 결제할 수 있다.

**Acceptance Criteria**:

1. Given 현재 세션에 주문이 존재할 때,
   When 주문 내역 화면에서 "분할 계산" 버튼을 누르면,
   Then 메뉴별/인원별 분할 모드 선택 화면이 표시된다.

2. Given 메뉴별 분할 모드를 선택했을 때,
   When 메뉴 항목을 체크박스로 선택하여 그룹A에 배정하면,
   Then 그룹A의 소계와 나머지 그룹의 소계가 실시간으로 계산되어 표시된다.

3. Given 인원별 분할 모드를 선택했을 때,
   When 인원 수(N)를 입력하면,
   Then 총 금액을 N으로 나눈 1인당 금액이 표시된다.

4. Given 분할 계산 결과가 표시되었을 때,
   When 화면을 확인하면,
   Then 분할 결과는 화면 표시 전용이며 서버에 저장되지 않는다.

---

### US-06: 주문 내역 조회
**우선순위**: P1
**페르소나**: 디지털 친숙 고객, 디지털 비친숙 고객
**요구사항**: FR-05

> As a 고객,
> I want 현재 테이블에서 주문한 내역을 확인할 수 있기를,
> So that 주문 상태와 총 금액을 파악할 수 있다.

**Acceptance Criteria**:

1. Given 고객이 주문 내역 화면에 접근했을 때,
   When 화면이 로드되면,
   Then 현재 테이블 세션의 주문만 시간 순으로 표시된다.

2. Given 주문 내역이 표시되어 있을 때,
   When 각 주문을 확인하면,
   Then 주문 번호, 주문 시각, 메뉴/수량, 금액, 상태(대기중/준비중/완료)가 표시된다.

3. Given 이전 세션(이용 완료 처리된)의 주문이 있을 때,
   When 주문 내역 화면을 조회하면,
   Then 이전 세션의 주문은 표시되지 않는다.

## 관리자 여정: 로그인 → 주문 모니터링 → 상태 관리 → 테이블 관리 → 메뉴 관리

### US-07: 매장 관리자 로그인
**우선순위**: P1
**페르소나**: 매장 관리자
**요구사항**: FR-06

> As a 매장 관리자,
> I want 매장 식별자와 계정 정보로 관리 시스템에 로그인할 수 있기를,
> So that 매장의 주문과 메뉴를 관리할 수 있다.

**Acceptance Criteria**:

1. Given 관리자가 로그인 화면에 있을 때,
   When 매장 식별자, 사용자명, 비밀번호를 입력하고 로그인 버튼을 누르면,
   Then JWT 토큰이 발급되고 주문 모니터링 대시보드로 이동한다.

2. Given 관리자가 로그인된 상태일 때,
   When 16시간이 경과하면,
   Then 자동으로 로그아웃되고 로그인 화면으로 이동한다.

3. Given 관리자가 로그인된 상태일 때,
   When 브라우저를 새로고침하면,
   Then 세션이 유지되어 로그인 상태가 유지된다.

4. Given 잘못된 인증 정보로 로그인을 시도할 때,
   When 연속으로 실패하면,
   Then 로그인 시도가 제한되고 안내 메시지가 표시된다.

---

### US-08: 실시간 주문 모니터링
**우선순위**: P1
**페르소나**: 매장 관리자
**요구사항**: FR-07

> As a 매장 관리자,
> I want 들어오는 주문을 실시간으로 테이블별 대시보드에서 확인할 수 있기를,
> So that 신속하게 주문을 처리할 수 있다.

**Acceptance Criteria**:

1. Given 관리자가 주문 모니터링 대시보드에 있을 때,
   When 고객이 새 주문을 생성하면,
   Then 2초 이내에 해당 테이블 카드에 신규 주문이 표시된다 (SSE).

2. Given 대시보드에 테이블 카드가 표시되어 있을 때,
   When 각 테이블 카드를 확인하면,
   Then 테이블 번호, 총 주문액, 최신 주문 미리보기가 표시된다.

3. Given 신규 주문이 도착했을 때,
   When 대시보드가 업데이트되면,
   Then 해당 테이블 카드가 시각적으로 강조된다 (색상 변경/애니메이션).

4. Given 테이블 카드가 표시되어 있을 때,
   When 테이블 카드를 클릭하면,
   Then 해당 테이블의 전체 주문 메뉴 목록 상세 보기가 표시된다.

---

### US-09: 주문 상태 변경
**우선순위**: P1
**페르소나**: 매장 관리자
**요구사항**: FR-07

> As a 매장 관리자,
> I want 주문의 상태를 변경할 수 있기를,
> So that 주문 처리 진행 상황을 추적하고 고객에게 알릴 수 있다.

**Acceptance Criteria**:

1. Given 주문 상세 보기에서 주문이 "대기중" 상태일 때,
   When "준비중"으로 상태 변경 버튼을 누르면,
   Then 주문 상태가 "준비중"으로 변경되고 대시보드에 반영된다.

2. Given 주문이 "준비중" 상태일 때,
   When "완료"로 상태 변경 버튼을 누르면,
   Then 주문 상태가 "완료"로 변경되고 대시보드에 반영된다.

3. Given 주문 상태가 변경되었을 때,
   When 고객이 주문 내역 화면을 조회하면,
   Then 변경된 상태가 반영되어 표시된다.

---

### US-10: 테이블 초기 설정
**우선순위**: P1
**페르소나**: 매장 관리자
**요구사항**: FR-08

> As a 매장 관리자,
> I want 테이블 태블릿의 초기 설정을 할 수 있기를,
> So that 고객이 해당 테이블에서 자동 로그인하여 주문할 수 있다.

**Acceptance Criteria**:

1. Given 관리자가 테이블 관리 화면에 있을 때,
   When 테이블 번호와 비밀번호를 입력하고 설정 버튼을 누르면,
   Then 테이블이 등록되고 16시간 세션이 생성된다.

2. Given 테이블이 설정 완료되었을 때,
   When 해당 테이블의 태블릿에서 서비스에 접속하면,
   Then 설정된 정보로 자동 로그인이 활성화된다.

---

### US-11: 주문 삭제
**우선순위**: P2
**페르소나**: 매장 관리자
**요구사항**: FR-08

> As a 매장 관리자,
> I want 잘못된 주문을 삭제할 수 있기를,
> So that 정확한 주문 현황과 금액을 유지할 수 있다.

**Acceptance Criteria**:

1. Given 테이블 상세 보기에서 주문 목록이 표시되어 있을 때,
   When 특정 주문의 "삭제" 버튼을 누르면,
   Then 확인 팝업이 표시된다.

2. Given 삭제 확인 팝업이 표시되어 있을 때,
   When "확인" 버튼을 누르면,
   Then 주문이 즉시 삭제되고 테이블 총 주문액이 재계산된다.

3. Given 주문 삭제가 완료되었을 때,
   When 대시보드를 확인하면,
   Then 삭제된 주문이 목록에서 제거되고 성공 피드백이 표시된다.

### US-12: 테이블 이용 완료
**우선순위**: P1
**페르소나**: 매장 관리자
**요구사항**: FR-08

> As a 매장 관리자,
> I want 고객의 식사가 끝나면 테이블 이용 완료 처리를 할 수 있기를,
> So that 다음 고객이 이전 주문 없이 새로 시작할 수 있다.

**Acceptance Criteria**:

1. Given 테이블에 활성 세션이 있을 때,
   When "이용 완료" 버튼을 누르면,
   Then 확인 팝업이 표시된다.

2. Given 이용 완료 확인 팝업에서,
   When "확인" 버튼을 누르면,
   Then 현재 세션의 주문 내역이 과거 이력으로 이동한다.

3. Given 이용 완료 처리가 완료되었을 때,
   When 대시보드를 확인하면,
   Then 해당 테이블의 현재 주문 목록과 총 주문액이 0으로 리셋된다.

4. Given 이용 완료 처리된 테이블에서,
   When 새 고객이 첫 주문을 생성하면,
   Then 새로운 세션이 시작되고 이전 주문 내역은 표시되지 않는다.

### US-13: 과거 주문 내역 조회
**우선순위**: P2
**페르소나**: 매장 관리자
**요구사항**: FR-08

> As a 매장 관리자,
> I want 테이블별 과거 주문 내역을 조회할 수 있기를,
> So that 이전 매출과 주문 이력을 확인할 수 있다.

**Acceptance Criteria**:

1. Given 테이블 상세 보기에서,
   When "과거 내역" 버튼을 누르면,
   Then 해당 테이블의 과거 주문 목록이 시간 역순으로 표시된다.

2. Given 과거 주문 내역이 표시되어 있을 때,
   When 각 주문을 확인하면,
   Then 주문 번호, 시각, 메뉴 목록, 총 금액, 이용 완료 시각이 표시된다.

3. Given 과거 주문 내역이 표시되어 있을 때,
   When 날짜 필터를 적용하면,
   Then 선택한 날짜 범위의 주문만 표시된다.

4. Given 과거 주문 내역 화면에서,
   When "닫기" 버튼을 누르면,
   Then 대시보드로 복귀한다.

### US-14: 메뉴 관리
**우선순위**: P1
**페르소나**: 매장 관리자
**요구사항**: FR-09

> As a 매장 관리자,
> I want 메뉴를 등록, 수정, 삭제하고 노출 순서를 조정할 수 있기를,
> So that 고객에게 최신 메뉴 정보를 제공할 수 있다.

**Acceptance Criteria**:

1. Given 관리자가 메뉴 관리 화면에 있을 때,
   When 카테고리를 선택하면,
   Then 해당 카테고리의 메뉴 목록이 표시된다.

2. Given 메뉴 등록 화면에서,
   When 메뉴명, 가격, 설명, 카테고리를 입력하고 이미지를 업로드한 후 저장 버튼을 누르면,
   Then 이미지가 S3에 업로드되고 메뉴가 등록되어 고객 화면에 반영된다.

3. Given 메뉴 목록에서 특정 메뉴를 선택했을 때,
   When 정보를 수정하고 저장 버튼을 누르면,
   Then 변경 사항이 저장되고 고객 화면에 반영된다.

4. Given 메뉴 목록에서 특정 메뉴를 선택했을 때,
   When 삭제 버튼을 누르면,
   Then 확인 후 메뉴가 삭제되고 고객 화면에서 제거된다.

5. Given 메뉴 등록 시,
   When 필수 필드가 비어있거나 가격이 유효 범위를 벗어나면,
   Then 검증 에러 메시지가 표시되고 저장이 차단된다.

6. Given 메뉴 목록이 표시되어 있을 때,
   When 메뉴의 노출 순서를 변경하면,
   Then 변경된 순서가 저장되고 고객 화면에 반영된다.

### US-15: 매장 관리자 계정 관리
**우선순위**: P1
**페르소나**: 슈퍼 관리자
**요구사항**: FR-10

> As a 슈퍼 관리자,
> I want 매장 관리자 계정을 생성, 수정, 삭제할 수 있기를,
> So that 각 매장에 적절한 관리자를 할당하고 관리할 수 있다.

**Acceptance Criteria**:

1. Given 슈퍼 관리자가 계정 관리 화면에 있을 때,
   When 매장을 선택하고 관리자 정보(사용자명, 비밀번호)를 입력하여 생성 버튼을 누르면,
   Then 매장 관리자 계정이 생성되고 활동 이력에 기록된다.

2. Given 매장 관리자 목록이 표시되어 있을 때,
   When 특정 관리자의 정보를 수정하고 저장하면,
   Then 변경 사항이 저장된다.

3. Given 매장 관리자 목록이 표시되어 있을 때,
   When 특정 관리자의 삭제 버튼을 누르고 확인하면,
   Then 계정이 삭제되고 활동 이력에 기록된다.

4. Given 슈퍼 관리자가 계정 관리 화면에 있을 때,
   When 매장별 관리자 현황을 조회하면,
   Then 각 매장에 할당된 관리자 목록이 표시된다.

### US-16: 슈퍼 관리자 활동 이력 조회
**우선순위**: P2
**페르소나**: 슈퍼 관리자
**요구사항**: FR-11

> As a 슈퍼 관리자,
> I want 계정 생성/삭제 활동 이력을 조회할 수 있기를,
> So that 감사 추적과 보안 관리를 수행할 수 있다.

**Acceptance Criteria**:

1. Given 슈퍼 관리자가 활동 이력 화면에 있을 때,
   When 화면이 로드되면,
   Then 모든 계정 생성/삭제 이력이 시간 역순으로 표시된다 (수행자, 대상 계정, 매장, 시각, 액션 유형).

2. Given 활동 이력이 표시되어 있을 때,
   When 날짜 필터를 적용하면,
   Then 선택한 날짜 범위의 이력만 표시된다.

3. Given 활동 이력이 표시되어 있을 때,
   When 액션 유형 필터(생성/삭제)를 적용하면,
   Then 선택한 유형의 이력만 표시된다.

4. Given 활동 이력 데이터가 존재할 때,
   When 이력 삭제를 시도하면,
   Then 삭제가 불가능하다 (append-only 정책).
